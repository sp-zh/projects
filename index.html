Algorithms, Principles, Formulas, and Code

1. PID Control

PID is the baseline feedback controller. It drives the tracking error e(t)=r(t)-y(t) toward zero using proportional correction, accumulated error, and predicted trend from the error derivative.

u(t)=K_p e(t)+K_i\int_0^t e(\tau)d\tau+K_d\frac{de(t)}{dt}

Proportional action gives immediate stiffness, integral action removes steady-state offset, and derivative action adds damping. The main practical issues are actuator saturation, integral windup, derivative noise amplification, and degraded behavior when the plant dynamics change. 

class PID:
    def __init__(self, kp, ki, kd, dt, u_min=-float("inf"), u_max=float("inf")):
        self.kp, self.ki, self.kd, self.dt = kp, ki, kd, dt
        self.u_min, self.u_max = u_min, u_max
        self.integral = 0.0
        self.prev_error = 0.0

    def update(self, reference, measurement):
        error = reference - measurement
        derivative = (error - self.prev_error) / self.dt
        candidate_integral = self.integral + error * self.dt
        u_unsat = self.kp * error + self.ki * candidate_integral + self.kd * derivative
        u = max(self.u_min, min(self.u_max, u_unsat))
        if u == u_unsat:
            self.integral = candidate_integral
        self.prev_error = error
        return u

2. Feedforward + PID

Feedforward uses a model of the plant or load to predict the input needed before error appears. PID then corrects modeling error, disturbances, and sensor bias.

u(t)=u_{ff}(r,\dot r,\ddot r)+K_p e+K_i\int e\,dt+K_d\dot e

For a mass-damper actuator, a common feedforward term is u_ff=m a_ref+b v_ref. This improves tracking and reduces feedback burden, but it depends on model accuracy and can command impossible inputs if saturation is ignored. 

def feedforward_pid(r, v_ref, a_ref, y, pid, mass, damping):
    u_ff = mass * a_ref + damping * v_ref
    u_fb = pid.update(r, y)
    return u_ff + u_fb

3. Notch, Lead, and Lag Compensation

Frequency-domain compensators shape loop gain. A notch attenuates a narrow resonance, a lead network adds phase margin and bandwidth, and a lag network increases low-frequency gain to reduce steady-state error.

G_{notch}(s)=\frac{s^2+2\zeta_z\omega_n s+\omega_n^2}{s^2+2\zeta_p\omega_n s+\omega_n^2},\quad \zeta_z<\zeta_p
G_{lead}(s)=K\frac{\tau s+1}{\alpha\tau s+1},\quad 0<\alpha<1

These methods are valuable when resonance, phase margin, or low-frequency tracking is the dominant issue. Their weakness is limited multivariable handling and sensitivity to resonance frequency drift. 

from scipy import signal

def notch_filter(omega_n, zeta_zero, zeta_pole):
    num = [1.0, 2.0 * zeta_zero * omega_n, omega_n**2]
    den = [1.0, 2.0 * zeta_pole * omega_n, omega_n**2]
    return signal.TransferFunction(num, den)

4. Gain Scheduling

Gain scheduling switches or interpolates controller parameters as operating conditions change. It is common when one linear controller cannot cover a wide nonlinear envelope, such as aircraft speed, motor load, battery voltage, or robot configuration.

K(\rho)=\sum_i w_i(\rho)K_i,\quad \sum_i w_i(\rho)=1,\quad w_i(\rho)\ge 0

The schedule variable ρ must be measurable or estimable. Poor scheduling can introduce discontinuities, hidden instability between design points, and unexpected behavior during fast transients. 

def scheduled_gain(speed):
    breakpoints = [0.0, 10.0, 25.0]
    kp_values = [3.0, 2.1, 1.2]
    if speed <= breakpoints[0]:
        return kp_values[0]
    if speed >= breakpoints[-1]:
        return kp_values[-1]
    for i in range(len(breakpoints) - 1):
        lo, hi = breakpoints[i], breakpoints[i + 1]
        if lo <= speed <= hi:
            a = (speed - lo) / (hi - lo)
            return (1 - a) * kp_values[i] + a * kp_values[i + 1]

5. Pole Placement

Pole placement designs state feedback so the closed-loop poles match desired locations. For a controllable linear system x_dot=Ax+Bu, choose u=-Kx+r so that the eigenvalues of A-BK determine the transient response.

\lambda(A-BK)=\{p_1,p_2,\ldots,p_n\}

It is transparent and effective for low-order systems, but aggressive pole choices can require large actuator effort and amplify unmodeled dynamics. 

import numpy as np
from scipy.signal import place_poles

A = np.array([[0.0, 1.0], [-2.0, -0.4]])
B = np.array([[0.0], [1.0]])
desired_poles = [-2.0 + 1.5j, -2.0 - 1.5j]
K = place_poles(A, B, desired_poles).gain_matrix

6. Linear Quadratic Regulator

LQR computes a state-feedback law that minimizes a quadratic trade-off between state error and control effort. It replaces manual pole selection with an optimization problem.

J=\int_0^\infty (x^TQx+u^TRu)\,dt,\quad u=-Kx,\quad K=R^{-1}B^TP
A^TP+PA-PBR^{-1}B^TP+Q=0

Larger entries in Q penalize state deviation; larger entries in R penalize actuator usage. LQR is powerful but assumes a reliable linear model and full-state feedback unless paired with an observer. 

import numpy as np
from scipy.linalg import solve_continuous_are

def lqr(A, B, Q, R):
    P = solve_continuous_are(A, B, Q, R)
    K = np.linalg.inv(R) @ B.T @ P
    return K

7. Linear Quadratic Integral Control

LQI augments LQR with integral states so the controller rejects constant disturbances and tracks references without steady-state error.

\dot z=r-Cx,\quad \frac{d}{dt}\begin{bmatrix}x\\z\end{bmatrix}=\begin{bmatrix}A&0\\-C&0\end{bmatrix}\begin{bmatrix}x\\z\end{bmatrix}+\begin{bmatrix}B\\0\end{bmatrix}u+\begin{bmatrix}0\\I\end{bmatrix}r

It is often the practical upgrade from LQR for tracking problems. The cost is slower dynamics, extra tuning, and windup-like behavior if saturation is not handled. 

def augment_for_lqi(A, B, C):
    n, m = B.shape
    p = C.shape[0]
    A_aug = np.block([[A, np.zeros((n, p))], [-C, np.zeros((p, p))]])
    B_aug = np.vstack([B, np.zeros((p, m))])
    return A_aug, B_aug

8. Kalman Filter + LQR

When all states are not measured, a Kalman filter estimates them from noisy measurements. LQR then acts on the estimate. Together they form the classic Linear Quadratic Gaussian controller under Gaussian noise assumptions.

\hat x^-_k=A\hat x_{k-1}+Bu_{k-1},\quad K_f=P^-C^T(CP^-C^T+R_v)^{-1}
\hat x_k=\hat x^-_k+K_f(y_k-C\hat x^-_k),\quad u_k=-K\hat x_k

The separation principle allows estimator and controller design separately for ideal linear systems. In real plants, poor noise models, delays, and unmodeled bias can destabilize or degrade performance. 

def kalman_update(x, P, u, y, A, B, C, Qn, Rn):
    x_pred = A @ x + B @ u
    P_pred = A @ P @ A.T + Qn
    S = C @ P_pred @ C.T + Rn
    L = P_pred @ C.T @ np.linalg.inv(S)
    x_new = x_pred + L @ (y - C @ x_pred)
    P_new = (np.eye(P.shape[0]) - L @ C) @ P_pred
    return x_new, P_new

9. Model Predictive Control

MPC repeatedly solves a finite-horizon constrained optimization problem. It predicts future states, chooses an input sequence, applies only the first input, then replans at the next sample.

\min_{u_{0:N-1}}\sum_{k=0}^{N-1}(x_k^TQx_k+u_k^TRu_k)+x_N^TP_fx_N
x_{k+1}=Ax_k+Bu_k,\quad x_k\in\mathcal X,\quad u_k\in\mathcal U

MPC is excellent for constraints, multivariable coupling, and economic objectives. Its issues are computational cost, model dependence, feasibility, and the need for robust fallback behavior. 

# Minimal cvxpy-style MPC sketch
import cvxpy as cp

def mpc_control(A, B, x0, Q, R, u_max, horizon):
    n, m = B.shape
    x = cp.Variable((n, horizon + 1))
    u = cp.Variable((m, horizon))
    cost = 0
    constraints = [x[:, 0] == x0]
    for k in range(horizon):
        cost += cp.quad_form(x[:, k], Q) + cp.quad_form(u[:, k], R)
        constraints += [x[:, k + 1] == A @ x[:, k] + B @ u[:, k]]
        constraints += [cp.abs(u[:, k]) <= u_max]
    problem = cp.Problem(cp.Minimize(cost), constraints)
    problem.solve()
    return u[:, 0].value

10. H∞ Control

H∞ control designs a controller that minimizes worst-case amplification from exogenous disturbances to regulated outputs. Instead of optimizing average quadratic performance, it optimizes a frequency-domain worst-case bound.

\|T_{zw}(s)\|_\infty=\sup_\omega \bar\sigma(T_{zw}(j\omega))<\gamma

Weighting functions encode performance, control effort, noise rejection, and robustness. H∞ is valuable when guaranteed disturbance rejection matters, but synthesis can produce high-order controllers and requires expert weighting. 

# Python-control sketch; exact APIs depend on installed slycot/control versions.
import control as ct

P = ct.ss(A, B, C, D)
# K, closed_loop, gamma, info = ct.hinfsyn(P, nmeas=1, ncon=1)

11. μ-Synthesis

μ-synthesis extends robust control to structured uncertainty. Instead of asking whether all uncertainty is small in an unstructured norm, it preserves the block structure of parameter variations, neglected dynamics, and actuator uncertainty.

\mu_\Delta(M)=\frac{1}{\min\{\bar\sigma(\Delta):\det(I-M\Delta)=0\}}

The standard workflow is D-K iteration: synthesize an H∞ controller for a scaled plant, analyze structured singular value, update scaling, and repeat. It is powerful for aerospace and precision systems, but hard to tune and may produce controllers too complex for embedded deployment. 

# Pseudocode for D-K iteration
K = initial_hinf_controller(P)
for _ in range(max_iterations):
    mu_bound, D_scale = structured_singular_value_analysis(P, K)
    P_scaled = apply_d_scaling(P, D_scale)
    K = hinf_synthesis(P_scaled)
    if mu_bound < 1.0:
        break

12. Reinforcement Learning Control

RL learns a policy from interaction. The controller chooses actions to maximize expected cumulative reward, often without an explicit analytic model. In control, the reward usually penalizes tracking error, energy, constraint violation, and unsafe states.

J(\pi)=\mathbb E_\pi\left[\sum_{t=0}^{\infty}\gamma^t r(s_t,a_t)\right],\quad a_t=\pi_\theta(s_t)
\nabla_\theta J\approx\mathbb E[\nabla_\theta\log\pi_\theta(a_t|s_t)A_t]

RL is attractive for high-dimensional nonlinear tasks and policies that are hard to hand-design. The practical problems are sample cost, safety during exploration, sim-to-real transfer, reward hacking, weak guarantees, and difficult debugging. 

# Minimal policy-gradient skeleton
for episode in range(num_episodes):
    states, actions, rewards = rollout(env, policy)
    returns = discounted_returns(rewards, gamma=0.99)
    loss = 0.0
    for s, a, G in zip(states, actions, returns):
        loss -= policy.log_prob(s, a) * G
    optimizer.zero_grad()
    loss.backward()
    optimizer.step()

Practical selection

How to Choose

Start simple

If the plant is single-input/single-output and constraints are mild, start with PID, then add feedforward, anti-windup, filtering, and frequency compensation only when the data says they are needed.

Use state-space when interactions matter

For coupled dynamics, multivariable plants, and measurable or estimable states, pole placement, LQR, LQI, and Kalman-filtered LQR offer a cleaner design language than many independent PID loops.

Use MPC when constraints are first-class

If actuator limits, safety envelopes, rate limits, and economic objectives dominate performance, MPC is often the most direct formulation.

Use robust or learning control deliberately

H∞ and μ-synthesis are justified when uncertainty must be certified. RL is justified when model-based design cannot capture the task and safe training infrastructure exists.
